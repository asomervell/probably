## ✨ Features

### Multi-Provider Bank Connections

Probably now supports connecting bank accounts through multiple secure providers, giving you flexibility and coverage across different banks and regions.

**What's New:**
- **Provider-agnostic architecture**: Unified system supporting Teller, Plaid, and Akahu
- **Teller integration**: Fully implemented and working (existing functionality preserved)
- **Plaid support**: Infrastructure ready (coming soon)
- **Akahu support**: Infrastructure ready for New Zealand banks (coming soon)
- **Unified connection interface**: Same experience regardless of provider
- **Automatic provider selection**: System chooses the best provider for your bank
- **Multi-provider accounts**: Connect accounts from different providers simultaneously

**Why This Matters:**
This foundational change enables Probably to support banks worldwide through multiple providers. You're no longer limited to Teller's coverage - when Plaid and Akahu are fully implemented, you'll be able to connect accounts from thousands of additional financial institutions. The unified interface means you get the same great experience regardless of which provider your bank uses.

**Technical Details:**
- New `Provider` interface in `internal/sync/providers/types.go`
- Provider registry system for managing multiple providers
- Database migration 041 adds provider-agnostic columns
- Migration 042 removes legacy Teller-specific columns (after data migration)
- All SQL queries updated to use new provider-agnostic columns
- Backward compatibility maintained through field mapping
- See [Multi-Provider Bank Connections](../features/bank-connections.md) for full documentation

**Migration Notes:**
- Existing Teller accounts are automatically migrated to the new schema
- No user action required - migration happens seamlessly
- All existing functionality continues to work
- Run migration 041, then the data migration script, then migration 042 when ready

**Commits:** `0ac1d6f`

### AI Chat Assistant

Natural language interface for querying your financial data with automatic SQL generation and execution.

**What's New:**
- **Conversational interface**: Ask questions in natural language
- **Automatic SQL generation**: AI generates and executes SQL queries
- **Query result caching**: Fast responses for repeated queries
- **Advanced visualizations**: Tables, charts, and natural language summaries
- **Suggested questions**: AI-generated question suggestions based on your data
- **Conversation context**: Follow-up questions understand previous context
- **Security hardening**: SQL injection prevention and query validation
- **HTMX + SSE**: Real-time streaming responses

**Why This Matters:**
Instead of learning SQL or navigating complex reports, you can now ask questions like "How much did I spend on groceries last month?" and get instant answers. The AI handles all the complexity of generating the right SQL query, executing it safely, and presenting results in an easy-to-understand format.

**Technical Details:**
- New `/chat` page with conversational UI
- SQL generation using orchestrator with tool calling
- Query optimization with execution plan analysis
- Result caching with TTL-based invalidation
- Frontend built with HTMX and Server-Sent Events
- See [AI Chat](../features/ai-chat.md) for full documentation

### Multi-Model Orchestrator

Advanced AI system that intelligently routes and executes LLM tasks using multiple models and execution strategies.

**What's New:**
- **Multiple execution strategies**: Simple, Escalate, Consensus, Supervisor, Verify
- **Model role system**: Fast, Reasoning, Tool Call, Verifier, Vision, Planner
- **Cost optimization**: Escalate strategy uses cheap models first, expensive ones only when needed
- **Quality assurance**: Consensus and Verify strategies for high-accuracy tasks
- **Multi-provider support**: Google (Gemini), xAI (Grok), Groq
- **Task types**: Categorization, P2P categorization, Chat, Statement extraction

**Why This Matters:**
The orchestrator provides a unified framework for all AI-powered features, enabling cost optimization, quality assurance, and flexible execution patterns. Features like AI Chat, categorization, and insights all benefit from intelligent model selection and execution strategies.

**Technical Details:**
- New `internal/orchestrator` package
- Strategy pattern for execution approaches
- Model role assignment for optimal task routing
- See [Multi-Model Orchestrator](../features/multi-model-orchestrator.md) for full documentation

## Technical Details

### Database Migrations

- **Migration 041**: Adds provider-agnostic columns to `accounts` and `transactions` tables
- **Migration 042**: Removes legacy Teller-specific columns (run after data migration)
- **Data migration script**: `scripts/migrate-teller-to-provider.sql` for migrating existing Teller data

### Code Architecture

- **Provider interface**: `internal/sync/providers/types.go` defines the Provider interface
- **Provider registry**: `internal/sync/providers/registry.go` manages provider instances
- **Teller provider**: `internal/sync/providers/teller.go` wraps existing Teller implementation
- **Sync service**: `internal/sync/service.go` provides unified sync interface
- **Sync worker**: `internal/sync/worker.go` updated to be provider-agnostic

### Documentation

- New feature documentation: `docs/features/bank-connections.md`

## Migration Guide

If you have existing Teller-connected accounts:

1. **Run migration 041**: Adds new provider-agnostic columns
2. **Run data migration script**: `scripts/migrate-teller-to-provider.sql` to copy Teller data to new columns
3. **Verify migration**: Check that all accounts have `provider = 'teller'` and `external_account_id` set
4. **Run migration 042**: Removes old Teller-specific columns (only after verification)

**Note**: The code maintains backward compatibility, so you can run migrations gradually. Old Teller fields are automatically mapped from new fields in the model layer.
