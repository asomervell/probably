# v1.22.0

**Date:** January 27, 2025  
**Previous Version:** v1.21.2  
**Commit Range:** v1.21.2..v1.22.0

## Summary

This release adds comprehensive AI observability, cost tracking, and monitoring capabilities to Probably, enabling deep insights into LLM operations, tool executions, and agent workflows. All AI operations are now fully instrumented with Sentry AI Agent Monitoring, following industry-standard conventions for tracking generative AI systems.

## ✨ Features

### AI Cost Tracking

**Automatic Cost Calculation**
- Cost tracking for all AI models (Google Gemini, xAI Grok, Groq) with per-model pricing
- Automatic calculation based on token usage from API responses
- Costs displayed in Sentry's AI Models dashboard
- Supports reasoning tokens and cached tokens for accurate cost attribution

**Model Pricing**
- Google Gemini 2.5 Flash: $0.075/1M input, $0.30/1M output tokens
- Google Gemini 2.5 Pro: $1.25/1M input, $5.00/1M output tokens
- xAI Grok 4-1: $0.10/1M input, $0.30/1M output tokens
- Groq: Free tier pricing

### Comprehensive Sentry AI Agent Observability

The entire AI system has been instrumented with Sentry AI Agent Monitoring, following the `gen_ai.*` operation naming conventions. This provides complete visibility into all LLM operations, tool calls, agent invocations, and pipeline executions.

**LLM Request Tracking (`gen_ai.request`)**
- All 7 `callModel` functions in the orchestrator are now instrumented
- Tracks model name, provider, operation type, messages, and available tools
- Records token usage (input, output, cached, reasoning tokens)
- Captures response text, tool calls, and finish reasons
- Supports both streaming and non-streaming operations
- **NEW**: Automatic cost calculation and tracking

**Agent Invocation Tracking (`gen_ai.invoke_agent`)**
- Agent spans for all chat handlers with full user context
- Tracks agent name, model, input, and output
- Records confidence scores, iteration counts, and escalation paths
- Includes metadata about task types and strategies
- **NEW**: Structured logging for agent lifecycle events

**Tool Execution Tracking (`gen_ai.execute_tool`)**
- Spans for `search_entities` and `find_matching_purchase` tool calls
- Captures tool name, description, input arguments, and output
- Tracks tool execution duration and errors

**Pipeline/Workflow Tracking**
- Pipeline spans for all strategy executions in `simple.go`
- Tracks categorization, P2P categorization, chat, and chat-with-tools pipelines
- Records step counts and execution metadata

**Agent Handoff Tracking (`gen_ai.handoff`)**
- Spans for agent-to-agent transitions (prepared for future multi-agent workflows)

### Enhanced Logging Integration

**Sentry Structured Logging**
- Sentry structured logging enabled (`EnableLogs: true`)
- Logs automatically link to traces for better debugging
- Added logging to key AI operations (agent starts, request completion, errors)
- New `GetLogger()` helper for easy access to Sentry loggers

**Sentry Log Writer**
- New `SentryLogWriter` captures all application logs as Sentry breadcrumbs
- Automatic log level detection (error, warning, info, debug)
- Dual output: logs to stdout and Sentry simultaneously
- Configurable log categories for better organization

**Breadcrumb Tracking**
- Strategic breadcrumbs added throughout orchestrator execution
- Context-rich debugging information for AI operations
- Automatic capture of key decision points and state changes

### Enhanced Trace Correlation

**Background Worker Transactions**
- Background workers now create Sentry transactions to ensure all AI spans are sent
- All worker operations are properly linked in trace views
- 100% sample rate for local environment to ensure all traces are captured

**Improved Span Hierarchy**
- Fixed span naming to match Sentry AI Agent conventions
- Proper parent-child relationships for all spans
- Agent spans added to V2 chat handlers for complete trace coverage
- Diagnostic warnings when spans are created without parent contexts

### Panic Recovery and Error Handling

**Recovery Helpers**
- `RecoverWithSentry`: Captures panics in goroutines and reports to Sentry
- `RecoverAndLog`: Graceful panic recovery for background jobs
- Automatic error classification and tagging

**Transaction Starters**
- `StartTransaction` helper for background jobs and workers
- Enables full trace context for operations outside HTTP requests
- Proper trace ID propagation for distributed systems

### User Context Integration

- Automatic user context setting in chat handlers
- Ledger ID tagging for all AI operations
- Enhanced error correlation with user and ledger information

## Technical Details

### Observability Architecture

The observability system has been completely rewritten to follow Sentry's AI Agent Monitoring conventions:

- **Operation Names**: Uses standard `gen_ai.*` prefixes (request, invoke_agent, execute_tool, handoff)
- **Data Attributes**: Follows Sentry's expected attribute naming (`gen_ai.request.model`, `gen_ai.usage.input_tokens`, `gen_ai.usage.cost.*`, etc.)
- **Span Hierarchy**: Proper parent-child relationships for nested operations
- **Context Propagation**: Full context passing through all AI operation layers

### New Files
- `internal/observability/cost.go`: Cost calculation and model pricing

### Modified Files
- `cmd/server/main.go`: Added Sentry configuration logging, test log, and environment detection
- `internal/observability/ai.go`: Fixed span naming to match conventions, added cost tracking methods (`SetCost`, `SetCostDetailed`)
- `internal/observability/sentry.go`: Enabled logs (`EnableLogs: true`), added `GetLogger()` helper, improved diagnostics, 100% sample rate for local
- `internal/orchestrator/api.go`: Extract token usage from API responses, calculate and record costs automatically
- `internal/orchestrator/orchestrator.go`: Added structured logging to agent lifecycle
- `internal/llm/handler.go`: Added agent spans for V2 chat handlers (JSON and SSE)
- `internal/processing/worker.go`: Create transactions for background operations to ensure spans are sent

### Instrumented Components

**Orchestrator API (`internal/orchestrator/api.go`)**
- `callModel`: Main LLM call function with cost tracking
- `callVertexAIStreaming`: Streaming Vertex AI calls
- `callVertexAI`: Non-streaming Vertex AI calls with token usage extraction
- `callOpenAIAPI`: OpenAI-compatible API calls with token usage extraction
- `callChatToolsModel`: Chat with custom tools
- `callVertexAIChatToolsStreaming`: Streaming chat tools
- `callOpenAIChatTools`: OpenAI chat tools

**Strategy Executions (`internal/orchestrator/simple.go`)**
- `executeCategorize`: Transaction categorization pipeline
- `executeCategorizeP2P`: P2P transaction categorization
- `executeChat`: Chat completion pipeline
- `executeChatTools`: Chat with tool calling
- `executeSearchEntities`: Entity search tool
- `executeFindMatchingPurchase`: Purchase matching tool

**Chat Handlers**
- `internal/handlers/chat.go`: `chatAskSSE` with agent spans
- `internal/llm/handler.go`: V2 chat handlers (JSON and SSE) with agent spans

**Orchestrator Core (`internal/orchestrator/orchestrator.go`)**
- Task execution with agent spans and structured logging
- Breadcrumb tracking for debugging

### Backward Compatibility

The observability package maintains backward compatibility with deprecated function names:
- `StartAISpan` → `StartAIRequest`
- `StartToolCallSpan` → `StartTool`
- `StartPipelineSpan` → `StartPipeline`
- `StartAgentSpan` → `StartAgent`

All deprecated functions are aliased to the new implementations, ensuring existing code continues to work.

### Database Changes
No database migrations required.

## Configuration

No configuration changes are required. The observability system automatically uses existing Sentry configuration. Ensure your `SENTRY_DSN` environment variable is set to enable monitoring.

**For Cost Tracking:**
1. Ensure `SENTRY_DSN` is set in your environment
2. Set `SENTRY_ENVIRONMENT=local` for development (100% sample rate)
3. Costs will automatically appear in Sentry's AI Models dashboard

## Upgrade Notes

This is a non-breaking release. All changes are additive and maintain backward compatibility. Existing code will continue to work, and the new observability features will automatically start tracking AI operations once deployed.

Cost tracking and improved observability work automatically once Sentry is configured. No action required.
