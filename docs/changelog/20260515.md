### What I reasoned on

**PostHog audit — error tracking (status=active, last 30d):**

**PostHog audit — logs (cfo-cloud service, last 30d):**

All 117,550 logs are at `severity_text: info` — the OTEL bridge records everything from the standard `log` package at INFO regardless of actual level. The body format `"cfo-cloud YYYY/MM/DD HH:MM:SS <message> key=value..."` is the Go `log.Printf` format passed through `stdLogToSlogWriter`, which becomes the OTEL log body.

**Key signal found:**

Every hourly sync (at minute :10, every hour for 30+ days) emits:

```
provider sync complete trigger=scheduled status=partial_failure connections=10 transactions=N errors=4
```

- `errors=4` is `syncErrors` (non-disconnected failures) — consistent across all 30 days
- `transactions` slowly increases (57→64) as new transactions accumulate
- `connections=10` is stable — 10 total Teller connections
- No per-account error log entries visible in PostHog (the `slog.ErrorContext("account sync failed")` logs are present in `worker.go` but their body doesn't contain "error" as a keyword, so they weren't found by the "error" text search)

**Root cause analysis:**

Matched against commit history: this pattern is explained by two bugs fixed in PR #462 (`1c84911`) and PR #478 (`c898d70`):

1. `TellerAPIError.IsConnectionDisconnected()` only checked `enrollment.disconnected` code prefix, not HTTP 401. The 4 disconnected accounts return 401 → counted as transient `syncErrors`, never marked disconnected.
2. `SyncTransactions` silently discarded `ConnectionDisconnectedError` from `reconcileOpeningBalance`, causing the same accounts to fail every cycle.

**Both fixes are already on `main`** (remote main was updated to `c898d70` which is HEAD of this branch). The production service just needs to be deployed.

**Secondary observation:**

`enrichment complete ... skipped=2514 failed=0` every 10 minutes — 2514 transactions already enriched, 0 new to process. Normal steady-state behavior.
