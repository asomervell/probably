### What I reasoned on (this run)

Queried PostHog error tracking (0 active issues) and log-level errors over the last 7 days. The only error-level logs were from the April 25 burst already documented in prior runs. No new errors since 2026-04-26.

Sampled the two most frequent distinct error bodies from the April 25 burst and checked the **current codebase** to verify the primary fixes hold and identify any remaining gaps:

**Error A — `[OTEL] export error: Too many AI spans: 246 exceeds limit of 100`**
- Primary fix (PR #121): removed 4 redundant transport-layer child `StartOperation` spans, halving per-trace span count.
- Gap identified: `WithMaxExportBatchSize(50)` without `WithMaxQueueSize` means the SDK's default queue of 2048 spans can accumulate and flush as a single 246-span request on shutdown. Adding `WithMaxQueueSize(75)` caps the queue so any flush sends at most one 50-span batch — well under PostHog's 100 limit.
- **Linear:** HAM-3937

**Error B — `[Orchestrator] Task categorize failed … Function call is missing a thought_signature … position 2`**
- Primary fix (PR #122): `ThinkingBudget: genai.Ptr[int32](0)` explicitly suppresses thinking on tool-active turns for Gemini 3/2.5 models.
- Gap identified: some model variants (e.g. `gemini-3.1-flash-lite-preview`) do not fully honour `ThinkingBudget=0`, generating a `ThoughtSignature` for the first function call in a multi-call response but not the second. Passing these inconsistent raw parts back on the next turn triggers the 400. Adding a `thoughtSignaturesConsistent()` guard before storing `RawVertexParts` ensures we only preserve raw parts when all function-call parts have a consistent signature state (all signed or all unsigned). Inconsistent responses fall back to the reconstructed `ToolCalls` slice with no signatures.
- **Linear:** HAM-3938
