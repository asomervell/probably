# 2026-06-03 — Session 4: PostHog error root-cause + sync observability fixes

| `provider_sync_completed` status field | All `None` in PostHog — type collision |

## Reasoning

Deep-dived the `errors=5 partial_failure` baseline rather than accepting it.

**Root cause 1: Teller 403 not classified as disconnected.**
`IsConnectionDisconnected()` on `TellerAPIError` only handled 401. Teller returns 403 when an enrollment requires action (MFA expired, access revoked). Without 403 handling, 5 accounts failed every sync cycle for 30+ days as apparent application errors, never being marked disconnected and never stopping.

**Root cause 2: `syncErrors++` placed before transient check.**
Transient (5xx) errors incremented `syncErrors` before the guard, triggering `partial_failure` status even for provider outages that would self-resolve.

**Root cause 3: Worker used `IsTellerTransientError` for all providers.**
Akahu transient errors (`IsAkahuTransientError` existed but was never called from worker) were not suppressed. Confirmed main already had `IsProviderTransientError` from a prior session's squash-merged history — so this was already partially fixed.

**Root cause 4: PostHog `"status"` field inferred as Numeric.**
String values `"success"` / `"partial_failure"` silently dropped to `None`. PostHog had inferred the type as Numeric from early zero-value sends.

**Root cause 5: `error_details` sent as `[]string`.**
PostHog can't store array properties — needs a joined string.

**Root cause 6: `CaptureFailure` not producing `$exception` events.**
Server-side exceptions not appearing in PostHog since May 26. Per-account errors were invisible.

**Branch complexity note:** `fix/sync-error-observability` had 51-commit rebase conflict because the branch history diverged far from main (local `main` had accumulated commits from `claude/clever-knuth-6JIaq` dev session). Resolved by resetting the branch to `origin/main` and applying the 3 remaining fixes (which weren't already on main) as a single clean commit.
