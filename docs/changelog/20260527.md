# 2026-05-27 — PostHog Error Triage + Sync Hardening (session 8)

## Reasoning

Re-confirmed PostHog state: 0 active errors, 0 error/warn logs in 30 days. All 19 historical issues
remain resolved or suppressed. Stack traces in PostHog still reference `personal-cfo` module path —
production binary has not been updated yet.

Identified two actionable code improvements from error patterns:

### 1. `ProviderBalancePullError` noise (suppressed, 12 occurrences)

The `reconcileOpeningBalance` → `getTellerBalanceCents` call path was calling `CaptureFailure`
for any non-5xx `TellerAPIError`. This included 4xx responses (e.g. account type doesn't support
the `/balances` endpoint). These are provider-side errors, not application bugs, and should not
fire PostHog exceptions. Extended the "skip CaptureFailure" rule from `>= 500` to all `TellerAPIError`
responses during balance reconciliation. Non-Teller errors (DB, logic) still captured.

### 2. `TellerSyncService` allocated per-request (DRY + efficiency)

`newTellerSyncService()` was creating a new `TellerSyncService` (with new `AccountStore`,
`TransactionStore`, `AccountOwnerStore`, `TransferMatcher`) on every handler call. The `TellerClient`
(TLS) was already shared; the stores are stateless pool wrappers. Cached the full `TellerSyncService`
in `Handlers` struct, initialized at startup. Fixed a DRY violation in `TellerLinkAccount` that
bypassed the helper and called `sync.NewTellerSyncService(...)` directly.

### 3. `getOrCreateOpeningBalanceAccount` — N+1 query

Was calling `GetByLedgerID` (loads all accounts for a ledger) just to find one equity account.
Replaced with a single targeted `SELECT id FROM accounts WHERE ledger_id = $1 AND type = 'equity'
AND name = 'Opening Balance'`.
