### Root cause: `bash -eo pipefail` + `grep` on no-match exits CI

`.github/actions/semver/action.yml` and `.github/actions/release-notes/action.yml` both run under `shell: bash` (GitHub Actions default: `bash -eo pipefail`). Both had:

```bash
# semver/action.yml:35
LAST_TAG=$(git tag -l 'v[0-9]*.' | grep -E '...' | head -1)

# release-notes/action.yml:34
PREV_TAG=$(git tag -l 'v[0-9]*.' | grep -E '...' | grep -v "..." | head -1)
```

When the repo has no matching semver tags (the case here since deployment never succeeded), `grep` exits 1 (no matches), `pipefail` propagates that exit code through the command substitution, and the script exits immediately — before the downstream `[ -z "$LAST_TAG" ]` guard ever runs. This caused every push to `main` to fail silently at the semver step, explaining why production was never updated and PostHog telemetry has been silent 35+ days.

**Fix:** Added `|| true` to both pipeline expressions. The downstream empty-string guard already handles the empty-variable case correctly (starts at `v0.1.0`).

PR #158 had already identified this same fix. Closed #158 and applied the fix directly to this session's branch.
