## What I reasoned on

PostHog error tracking (project 88871, Pre-production) was checked for active and historical errors.
Found 35 total issues, all already marked **resolved**. They collapsed into **3 distinct root causes**, all originating from the `worker:probably-categorization` worker during a burst on 2026-04-25.

### Error type 1 — `function_response.name: Name cannot be empty` (666 occurrences, 7 issues)
PostHog representative: `019dc622-40ff-7771-8234-8d31fca0b72e`

Gemini's `GenerateContent` API rejects `FunctionResponse` parts with an empty `Name` field. This happened when tool call IDs were provider-assigned UUIDs that didn't carry the function name, and the message serialisation path didn't fall back to inferring the name from arguments or content.

**Already fixed** by PRs #104, #105, and `refactor: deduplicate Vertex AI message conversion` (f7a2e31):
- `sanitizeFunctionResponseNames()` patches any empty name to `"search_entities"` before the API call.
- `convertMessagesToVertexContents()` has a 4-level fallback: `ToolCallID → expectedNames queue → payload inference → "search_entities"`.
- `annotateToolResponse()` embeds `function_name` + `name` in every tool response payload.

### Error type 2 — `Function call is missing a thought_signature` (228 occurrences, 1 issue)
PostHog representative: `019dc625-4e4b-7053-a3fa-f35a036e9263`

Gemini thinking models require a `thought_signature` on every `FunctionCall` part when `ThinkingConfig` is enabled. If a prior turn generated function calls without thinking (e.g. because thinking was disabled for tools), re-enabling `ThinkingConfig` on the next turn causes a 400.

**Already fixed** by PRs #96, #102, #106:
- `hasToolHistory(messages)` returns true whenever any message in the thread is a tool response or carries tool calls.
- `callVertexAI` and `callVertexAIChatToolsStreaming` both gate `ThinkingConfig` on `!hasToolHistory(messages) && !includeTools`.
- Non-Vertex Gemini path: `allowToolCalls` is forced to `false`, preventing the model from emitting function calls that would lack thought signatures.
- Errors are classified as `ErrTypeParseError` / `retryable: false` so the worker doesn't hammer the API.

### Error type 3 — `Too many AI spans: N exceeds limit of 100` (26 occurrences, 27 issues)
PostHog representative: `019dc623-3ebf-73d0-b0d8-1e63e55d22ef`

PostHog's AI OTEL endpoint rejects OTLP trace batches containing more than 100 spans. Long categorisation jobs with many tool iterations exceeded this limit.

**Already fixed** by PR #104:
- `otel.go` sets `sdktrace.WithMaxExportBatchSize(50)`, keeping every exported batch well below the 100-span ceiling.

## Third run — 2026-04-27 (OTEL spans + thought_signature deep-dive)

### What I reasoned on (this run)

Read previous runs' context from this file. Queried PostHog logs for new errors. Found two error patterns requiring fixes in `internal/orchestrator/api.go`:

**Error E — `Too many AI spans: N exceeds limit of 100` (recurring)**

The `WithMaxExportBatchSize(50)` fix (PR #104) kept batch sizes below 100, but PostHog's limit is per-trace, not per-batch. Root cause: `processParallelBatches` in `worker.go` shares a single root trace across all concurrent LLM sub-batches. With 3 concurrent sub-batches × up to 5 tool-calling iterations × 2 spans per call (parent `gen_ai.*` + child transport span `ai.vertex_ai`/`ai.openai_api`), the total per-trace AI span count reached 200–246.

**Already fixed** by PR #121:
- Removed four redundant nested transport-layer child spans from `callVertexAI`, `callOpenAIAPI`, `callVertexAIChatToolsStreaming`, `callOpenAIChatTools`. Each function created a `StartOperation` span nested inside the parent `gen_ai.*` span, doubling the per-trace span count without adding observability.

**Error F — `Function call is missing a thought_signature` (recurring)**

`gemini-3.1-flash-lite-preview` is a Gemini 3 thinking model. When `ThinkingConfig` is `nil`, the model applies implicit thinking inconsistently — attaching a `thought_signature` to some function call parts but not others (e.g., position 2 in a multi-function response). On the next turn, Vertex AI rejects because function call history contains an unsigned part.

The existing `!hasToolHistory(messages) && !includeTools` guard correctly prevents enabling thinking after tool history exists, but with `ThinkingConfig = nil`, the model can still produce implicit thoughts unpredictably. Explicitly setting `ThinkingBudget: 0` suppresses all thought generation during tool turns.

**Already fixed** by PR #122:
- Changed `ThinkingConfig` logic in `callVertexAI` and `callVertexAIChatToolsStreaming` from a nil/enable toggle to an explicit two-state policy: enable thinking (with `IncludeThoughts: true`) only when no tools are active and no tool history exists; explicitly suppress thinking (with `ThinkingBudget: genai.Ptr[int32](0)`) in all other cases for thinking models.

Also addressed across runs #3–#6 (PRs #115–#124):
- **PR #115**: `SupportsVision()` guards non-Google providers; Plaid error detail surfacing; dead `executeParseStatement` removed.
- **PR #116**: `log.Printf` → slog in `orchestrator/{consensus,verify}`, `extraction/dedupe`, `chat/optimizer`.
- **PR #117**: `plaid_handlers.go` `log.Printf` → slog; Gemini `API_KEY_INVALID` classified as `[permission_denied]`.
- **PR #118**: Remaining ~347 `log.Printf` calls across 26 files migrated to slog.

- **PR #120**: Dead `parseExtractionResponse` (78 lines) removed; `slog.Error(fmt.Sprintf(...))` antipattern fixed in 4 sites.
