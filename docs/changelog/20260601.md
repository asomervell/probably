### Code defect found during investigation

While reading the current sync code to understand the analogous error patterns, found a genuine bug in `IsProviderTransientError` (`internal/sync/retry.go`):

```go
// BEFORE (buggy):
func IsProviderTransientError(err error) bool {
    return IsTellerTransientError(err) || IsAkahuTransientError(err) || IsPlaidTransientError(err, (*http.Response)(nil))
}
```

`IsPlaidTransientError(err, nil)` has a network catch-all that returns `true` for any error that is not a structured Plaid error AND has a nil httpResp. This means **Teller or Akahu 4xx errors** (400, 422, etc.) that aren't caught by `IsConnectionDisconnectedError` (handles 401/disconnected) or the 5xx transient checks would be silently swallowed as "transient" instead of being captured to PostHog.

## Reasoning (session 2)

- The PostHog issues are all historical and already resolved/suppressed. Cannot and need not "fix" them in PostHog.
- The code defect IS actionable: it prevents current-architecture sync failures from reaching PostHog, masking real errors.
- **Root cause**: `IsPlaidTransientError` doubles as a network-failure detector (returns `true` for unrecognised errors with nil httpResp), which is appropriate when called from within Plaid's own retry loop ‚Äî but incorrect when called multi-provider from `IsProviderTransientError`.
- **Fix**: Replace the delegating call with a direct `plaidExtractError` check that only matches structured Plaid error types (`API_ERROR`, `RATE_LIMIT_EXCEEDED`).
