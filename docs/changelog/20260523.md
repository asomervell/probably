# 2026-05-23 — Sync error investigation and fixes

## Context

Picked up from PostHog investigation. Scheduled sync runs consistently show
`status=partial_failure` with `errors=4` across multiple days. The same
connections fail every run. PostHog log view shows all records at
`severity_text: info / severity_number: 9` regardless of actual slog level —
ERROR-level slog records do not surface distinctly because HTTP access logs and
stdlib `log.Printf` calls pass through `stdlog_bridge.go` at INFO, and the
OTLP pipeline appears to normalise severity.

The actual "account sync failed" `slog.ErrorContext` calls were not visible in
PostHog search at all — could not see which accounts or what error was occurring.

## What was investigated

- PostHog `query-logs` for "error", "failed" — most results were INFO-level
  sync completion summaries; no per-account error detail was visible.
- `query-trends` on `provider_sync_completed` event — confirmed consistent
  `errors=4`, `status=partial_failure` across every sync run for days.
- Traced the sync pipeline: `SyncWorker.syncAll` → `SyncService.SyncTransactions`
  → provider-specific `SyncTransactions` → `syncTransactionsImpl` → impl.
- Read `stdlog_bridge.go`, `slog_multi.go`, `otel.go`, `main.go:initAppSlog`
  to understand why PostHog sees everything as INFO.
- Audited `GetAllWithProviderCredentials` SQL query for filtering gaps.

## What was fixed and shipped

### PR: `claude/clever-knuth-4KcEq`

1. **`noCredentialsError` in `providers/types.go`**
   - `syncTransactionsImpl` was returning `fmt.Errorf("no access token for account")`
     which caused `syncErrors++` in the worker. These accounts should be treated
     as disconnected (same as a revoked token), not app errors.
   - Added `noCredentialsError` implementing `ConnectionDisconnectedError`.
   - Effect: missing-credential accounts go to `markAccountDisconnected` path;
     they are counted under `disconnected`, not `syncErrors`.

2. **Exclude `login_required` from sync in `models/account.go`**
   - `GetAllWithProviderCredentials` excluded `disconnected` but not
     `login_required`. Accounts in `login_required` state fail every auth
     attempt and were contributing to persistent `errors=4`.
   - Added `AND connection_status IS DISTINCT FROM 'login_required'` to query.

3. **Per-account error detail logging in `worker.go`**
   - Added `syncErrorDetail` struct and per-account error collection.
   - Logs each failed account at INFO level (with provider, account name,
     account_id, connection_id, err) before the completion summary.
   - Workaround for PostHog not showing ERROR-level slog records distinctly.

4. **Remove dead access-token guards in `teller.go`, `akahu.go`, `plaid_sync.go`**
   - Each provider's `SyncTransactions` had an early `if accessToken == ""`
     check that `syncTransactionsImpl` already handles. Removed the dead branches.

## What was left behind

- **Root cause of 4 failing accounts unknown** — the new per-account INFO logs
  will appear in PostHog after the next scheduled sync (every 15 min). Check
  PostHog for `sync error detail` log lines to see actual error messages.
- **Linear MCP deprecated** — `mcp__Linear__*` tools rejected with
  "DEPRECATED TRANSPORT — REMOVAL DATE 2026-04-08 (past)". Cannot create
  Linear issues. All issue tracking done via GitHub PR description.
- **PostHog severity** — ERROR-level slog records still don't surface
  distinctly in PostHog log view. The INFO-level per-account error detail
  logging is a workaround; a proper fix would need the OTLP pipeline to
  correctly map slog levels to OTLP severity numbers. Not tackled here.
- **Dependabot alerts** — GitHub reports 1 high + 2 moderate vulnerabilities
  on the default branch. Not investigated.
