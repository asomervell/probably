### Root cause analysis

`syncAccountIdentity` (teller.go) had a non-atomic delete-then-create pattern:
- `DeleteByAccountID` succeeded ‚Üí all owners removed from DB
- Any subsequent `Create` in the loop failed ‚Üí **account left with zero owners** (silent data loss)
- If `DeleteByAccountID` itself failed ‚Üí code logged warning and continued ‚Üí **duplicate owner rows** possible after loop

This is a correctness issue, not currently surfacing errors (Teller identity sync is working), but would cause hard-to-debug data inconsistency if Postgres had transient write errors.

### Fix shipped

**PR #826** (`fix/sync-account-identity-atomic`, merged):

- `internal/models/account_owner.go`: Added `ReplaceForAccount(ctx, accountID, owners)` ‚Äî wraps delete + all creates in a single `pgx.BeginFunc` transaction. Either all owners are replaced or nothing changes.
- `internal/sync/teller.go`: `syncAccountIdentity` now calls `ReplaceForAccount`; owner-building logic extracted to `buildAccountOwners` (pure helper, no DB calls ‚Äî keeps the transaction scope minimal).

### What was ruled out this session

- **Ownership verification boilerplate** (`api_accounts.go`, `api_tags.go`, `api_rules.go`): GetByID + LedgerID check appears ~25√ó. Each resource returns a different type ‚Äî per-resource helpers reduce boilerplate marginally but add abstraction. Deferred (session 5 reasoning still holds).
