### What was confirmed

- However: deeper root-cause analysis of `errors=4` revealed a code bug independent of the deployment state.

### Root cause identified: HAM-5121

`TellerAPIError.IsConnectionDisconnected()` did not include `IsUnableToProcess()`. Teller returns "unable to process" for accounts needing re-authentication, but the sync worker's `providers.IsConnectionDisconnectedError()` check returned `false` for these, so they were counted as `syncErrors` (incrementing `errors=N`) instead of `disconnected`. The accounts were never marked disconnected in the DB and were retried every 15 minutes indefinitely.

Additionally, `internal/handlers/teller.go` had two redundant `IsUnableToProcess` checks that pre-dated the shared `ConnectionDisconnectedError` interface — dead code after the fix.

### What was fixed and shipped

**HAM-5121** — `sync: TellerAPIError 'unable to process' not treated as disconnected`

Files changed:
- `internal/sync/teller.go`: `IsConnectionDisconnected()` now returns `true` when `IsUnableToProcess()` is true (one-line addition)
- `internal/handlers/teller.go`: removed redundant `IsUnableToProcess` block from `syncSingleEnrollment` and simplified error classification in `TellerSyncMulti` (removed `var tellerAPIErr` + `errors.As` condition)

PR #561 merged to `main` (squash commit `299ac4ac`). HAM-5121 marked Done.

All tests passed: `go test ./internal/sync/... ./internal/handlers/...`
